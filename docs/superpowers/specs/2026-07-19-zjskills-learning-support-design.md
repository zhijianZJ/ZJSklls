# ZJSkills 学习陪跑与动态调整设计规范

## 背景与目标

ZJSkills 已能从目标、能力差距、项目证据和约束生成学习路线，并在测评失败或目标变化后重新规划。当前缺口发生在日常执行中：新手通常只会说“看不懂”“不会做”“报错了”“这周没完成”，现有工作流缺少一个低门槛入口，把这些自然语言问题转化为可执行的小步骤，并判断是否需要调整任务、周计划、路线或目标。

本次升级增加一个横向的学习陪跑闭环。它不是新的固定阶段，而是可以在 Weekly Planner 之后以及任何执行节点调用的支持能力。

## 设计原则

1. 默认使用混合模式：简单、低风险且上下文充分的问题直接给出一个下一步；复杂、信息不足或影响较大的问题进入逐步引导，每轮最多询问一个决定性问题。
2. 默认面向新手使用自然语言。结构化状态、Schema、版本和引擎名保留在内部；除非用户要求，不在首屏暴露。
3. 先定位最早阻塞点，再补知识或修改计划；不得把问题归因为懒惰、智力、人格或固定学习风格。
4. 每次输出一个可执行动作、明确完成标志和失败后的备用动作，避免一次性倾倒完整教程。
5. 使用最小影响原则调整计划：能解决当前问题时不重算周计划，能调整周计划时不重建路线；目标变化时必须回到 Goal Analysis。
6. 健康、心理、医疗、法律、财务和生产安全等高风险问题遵循既有安全边界，不冒充专业服务。

## 用户入口与路由

入口覆盖以下自然表达：

- “我不知道今天学什么。”
- “这部分看不懂。”
- “听懂了但是不会做。”
- “我遇到报错了。”
- “项目不知道怎么开始。”
- “这周没完成。”
- “最近时间变少了。”
- “目标变了。”
- “我不知道自己卡在哪里。”

系统先进行轻量路由：

- `direct_action`：问题清晰、风险低、所需上下文充分，直接给一个动作。
- `guided_diagnosis`：存在多个合理原因、缺少关键证据或用户无法描述卡点，每轮只问一个能区分路线的问题。
- `safety_handoff`：问题超出学习系统边界或具有高风险，给出安全说明和适当的人工/专业支持建议。

## 学习陪跑闭环

1. **听懂问题**：保留用户原话，并用一句通俗语言复述。
2. **定位卡点**：从九类问题中形成待验证判断：知识未理解、理解但不会实践、不知道如何开始、前置能力缺失、工具或环境故障、任务跨度过大、资源不适合、时间精力变化、目标方向变化。
3. **找到最早阻塞点**：优先寻找最早无法继续的步骤，而不是从最后的失败结果倒推人格结论。
4. **拆成最小动作**：给出当前可在短时间内完成、产生可观察结果的一步。
5. **定义完成标志**：明确用户如何判断这一步成功，不能使用“学会了”“理解了”等不可观察表述。
6. **执行与反馈**：用户完成后报告结果；失败时使用新证据继续拆解或切换方法。
7. **记录与关闭**：记录解决结果、证据和可复用经验；未解决的问题保持开放，不伪造结论。
8. **评估计划影响**：选择最小调整范围，并把需要重算的产物交给现有引擎。

## 新手默认回复卡片

默认自然语言回复按以下顺序呈现，空项不显示：

1. 你现在卡在哪里
2. 我对原因的初步判断
3. 现在只做这一步
4. 做到什么算完成
5. 如果还是不行
6. 是否影响原计划

复杂模式下，若关键证据不足，第三项改为一个决定性问题；不得同时给长教程和多个问题。用户说“不知道怎么回答”时，提供 2–3 个通俗选项或一个极小验证任务。

## 分级调整

计划影响使用五级枚举：

- `none`：只解决当前问题。
- `task`：替换、拆分、降阶或重新安排当前任务。
- `week`：容量、持续性或多项任务受影响，重新生成本周计划与最低交付。
- `roadmap`：前置依赖、项目难度或长期容量发生变化，重新检查路线图及受影响下游。
- `goal_system`：目标、期限、成功证据或职业方向发生变化，回到 Goal Analysis 并重建全部受影响内容。

一次偶发失败默认不能触发路线重建；连续相同障碍、真实约束变化或证据表明依赖设计错误时才能升级影响级别。

## 数据契约

新增 `learning-issue` 产物，支持跨轮次跟踪但不强制每次持久化。核心字段：

- 身份与版本：`id`、`schema_version`、`content_version`、`status`、时间、来源和置信度；
- 问题：`user_words`、`context`、`issue_type`、`observed_symptom`；
- 诊断：`suspected_causes`、`earliest_blocker`、`routing_mode`；
- 行动：`next_action`、`success_signal`、`fallback_action`；
- 结果：`resolution_status`、`result_observation`、`evidence_ids`；
- 影响：`plan_impact.level`、`reason`、`affected_artifacts`、`rollback_target`；
- 下一次交互：`next_check`。

验证器识别根目录 `learning-issue.yaml` 和 `learning-issues/*.yaml`。Schema 应拒绝缺少成功标志的动作、非法问题类型、非法影响等级，以及 `goal_system` 未指向 Goal Analysis 的记录。

## 集成边界

- `SKILL.md`：扩展触发描述和条件路由，保持入口简洁。
- `workflow.md`：定义横向支持循环及其与 canonical `engine_result` 的关系，不改变 11 个主阶段顺序。
- `problem-solving-engine.md`：保存完整诊断、拆解、验证与升级规则。
- `beginner-interaction.md`：保存自然语言入口、回复卡片和渐进披露规则。
- `planner-engine.md`：接收 `task` 与 `week` 级调整。
- `assessment-engine.md`：把解决结果作为观察，不自动升级为能力证据。
- `optimization-engine.md`：接收 `roadmap` 与 `goal_system` 级变更，并保持版本历史。
- `meta-learning-engine.md`：只有重复且有证据的学习方法问题才触发实验。

## 文档与测试

中英文 README、入门指南、完整手册和示例增加“学习中卡住了”的入口、示例对话、分级调整说明。测试覆盖：概念不懂、会说不会做、401 报错、不知道项目第一步、单周未完成、容量下降、目标变化、无法描述卡点。所有既有回归、Schema 校验、Markdown 链接和品牌中立测试必须继续通过。

## 验收标准

- 用户无需知道 Skill、引擎或 YAML 术语即可开始求助。
- 简单问题得到一个动作和成功标志；复杂问题每轮只有一个决定性问题。
- 每个问题都有可追踪状态、备用动作和计划影响判断。
- 系统只重算最小必要范围，目标变化除外。
- 运行时内容保持品牌和推广中立；ZJSkills 品牌仅出现在允许的元数据和公开文档。
