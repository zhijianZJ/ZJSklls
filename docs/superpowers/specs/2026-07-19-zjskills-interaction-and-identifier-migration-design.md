# ZJSkills 交互导航与技术标识迁移设计规范

## 1. 背景

ZJSkills 已具备学习方向判断、路线规划、周计划、问题拆解、阶段评估和动态优化能力，但当前用户必须先知道“该怎么提问”。对刚接触 AI、转行规划或 Agent Skills 的用户来说，空白输入框、专业术语和结构化状态仍然形成使用门槛。

同时，当前界面显示名已经是 `ZJSkills`，底层目录、Skill 名称和显式调用仍是 `learning-architect`。这导致安装路径、调用方式和品牌名称不一致。

本次升级解决两个耦合问题：建立跨平台可用的对话式交互导航；将底层技术标识统一迁移为符合 Agent Skills 命名规范的 `zjskills`。

## 2. 目标与非目标

### 2.1 目标

- 用户只说“我想学 AI”或“我不知道怎么开始”时，也能获得明确入口。
- 需求明确时直接进入相应能力，不强制先看菜单。
- 每轮默认只展示完成当前决策需要的信息，并提供清晰下一步。
- 支持自然语言和数字选择，不要求用户记住固定命令。
- 支持第一次使用、继续进度、今日/本周行动、学习卡点和条件变化五类高频任务。
- 正式使用 `zjskills/`、`name: zjskills`、`$zjskills` 和 `/zjskills`。
- 为旧安装提供清晰、可回滚的迁移指导，不保留会被重复发现的第二份运行时 Skill。

### 2.2 非目标

- 不开发独立网页、移动端、桌面 GUI 或平台专属插件。
- 不假设宿主支持真正可点击的按钮、表单或卡片组件。
- 不新增登录、账户、云同步或远程服务。
- 不在没有品牌素材的情况下自行设计图标、Logo 或品牌颜色。
- 不以交互优化为理由弱化证据、隐私、版本、闸门或安全规则。

## 3. 技术命名

Agent Skills 技术名称只能使用小写字母、数字和连字符。因此使用以下正式命名：

| 表面 | 正式值 |
|---|---|
| GitHub 仓库 | `zhijianZJ/ZJSkills` |
| 仓库运行时目录 | `zjskills/` |
| `SKILL.md` frontmatter | `name: zjskills` |
| Codex 显式调用 | `$zjskills` |
| Claude Code 显式调用 | `/zjskills` |
| UI 显示名 | `ZJSkills` |
| 测试目录 | `tests/zjskills/` |
| 发布版本 | `2.0.0` |

`ZJSkills` 只作为人类可见名称；`zjskills` 是跨平台技术标识。不得创建 `ZJSkills/` 大写目录，因为它违反 Skill 命名规则并在大小写不敏感文件系统上带来歧义。

## 4. 交互架构

### 4.1 组件边界

- `zjskills/SKILL.md`：只保留触发、引用路由、主工作流和交互入口规则。
- `zjskills/references/interaction-orchestrator.md`：负责首页导航、状态感知、意图路由、全局指令、回复结构和显示深度。
- `zjskills/references/beginner-interaction.md`：继续负责小白语言、单步引导、“不知道”恢复和学习卡点回复。
- 现有 Discovery、Planner、Problem-Solving、Optimization 等引擎继续负责业务决策，交互层不得复制或替代其规则。
- `zjskills/agents/openai.yaml`：提供 Skill 列表中的显示名、短说明和以 `$zjskills` 开头的默认提示词。

交互层是“导航和展示层”，业务引擎是“决策层”。交互层只选择正确引擎、压缩展示并提出下一步，不自行生成与引擎冲突的学习结论。

### 4.2 状态来源

交互层从已有状态派生界面，不新增独立的 interaction-state 文件：

- 是否存在有效的 `system-state.yaml`；
- 当前阶段及其闸门；
- 活动周计划和最低交付；
- 当前开放的 `learning-issue`；
- 最近一次下一动作；
- 用户明确选择的显示深度。

显示深度可作为学习者画像中的 `learning_preferences.interaction_depth` 保存，取值为 `beginner`、`standard` 或 `professional`。未设置时默认 `beginner`。没有写权限时只在当前对话中保持偏好，不声称已持久化。

## 5. 首次进入与意图路由

### 5.1 模糊请求

当用户只表达“想学 AI”“帮我看看”“不知道怎么开始”等无法确定路线的请求时，显示文本导航：

```text
ZJSkills 学习导航

你好，我会根据你的真实情况，帮你规划并陪你完成学习。

你现在最需要什么？

1. 了解 AI 行业，判断适合的方向
2. 制定 AI 学习或转行路线
3. 安排今天或本周学习任务
4. 解决学习中遇到的问题
5. 时间、目标或情况变了，调整原计划
6. 继续上次的学习进度

回复数字或直接说你的情况都可以。
不知道怎么选，可以回复“不知道”。
```

没有可读取的历史状态时，第 6 项应解释为需要用户提供或指定学习者工作区，不能假装已经保存进度。

### 5.2 明确请求

需求已经明确时不显示首页菜单，直接路由：

- 行业探索或方向不明 -> Discovery 与 Goal Analysis；
- 学习/转行路线 -> 现有 11 阶段工作流；
- 今天/本周学什么 -> Weekly Planner；
- 看不懂、不会做、报错、无法开始 -> Problem-Solving；
- 时间、预算、设备或目标变化 -> Problem-Solving 判级后进入 Planner、Roadmap 或 Optimization；
- 继续上次 -> 验证状态并从最早未通过闸门恢复。

用户回复数字、选项文本或自然语言同义表达时结果一致。选择与上下文冲突时，用一句话确认真正意图，不同时追问多个问题。

## 6. 状态感知界面

### 6.1 常规学习状态卡

每轮根据实际信息选择性展示，空项不显示：

```text
当前目标
目前进度
这次要解决什么
现在只做哪一步
做到什么算完成
完成后怎么继续
```

状态卡必须满足：

- “目前进度”依据阶段、产物或证据，不用主观百分比伪造精确度；
- “现在只做哪一步”只有一个主要行动；
- “做到什么算完成”必须可观察；
- “完成后怎么继续”最多给 3 个相关选择；
- 默认不显示引擎名、YAML、Schema、版本 ID 或内部置信度字段。

### 6.2 周进度卡

存在活动周计划时可显示：

```text
本周学习进度

已完成：[有证据的数量]
进行中：[数量]
遇到问题：[开放问题数量]
本周最低交付：[已完成 / 未完成 / 缺少证据]

建议：[一个决策]

下一步：
1. [最相关行动]
2. [查看或调整]
3. [暂停或返回]
```

没有证据时使用“未报告”或“缺少证据”，不能自动把计划项标记完成。

### 6.3 问题解决卡

沿用现有学习陪跑六项结构：卡点、初步判断、现在一步、完成标志、备用动作、计划影响。复杂问题的“现在一步”可以是一个决定性问题，但每轮只能有一个。

### 6.4 完成反馈卡

用户完成一步后，先确认可观察结果，再提供与情境相关的最多 4 个选项：继续、复习/独立重做、调整、暂停。不得在没有独立证据时把“在指导下完成”显示为能力已掌握。

## 7. 全局自然语言指令

以下表达在任意阶段均可识别，但不要求精确匹配：

| 意图 | 示例表达 | 行为 |
|---|---|---|
| 继续 | “继续”“下一步” | 从最近明确动作和有效状态继续 |
| 返回 | “返回上一步”“刚才那个” | 返回最近可逆的交互选择，不回滚已验证数据 |
| 换方法 | “换一种方式”“还是不懂” | 保留失败观察，切换表示或缩小步骤 |
| 不知道 | “不知道”“不会选” | 给 2–3 个通俗选项或一个极小验证任务 |
| 查看 | “查看完整计划”“为什么这样安排” | 提升本轮披露深度，不永久改变模式，除非用户要求 |
| 改条件 | “这周只有 4 小时”“设备变了” | 进入约束变化判级 |
| 改目标 | “我不求职了”“想创业” | 进入 `goal_system` 变更 |
| 保存 | “保存当前进度” | 仅在用户指定并授权工作区后写入和验证 |
| 暂停 | “先暂停”“以后继续” | 给出可恢复摘要，不虚构后台任务 |

“返回上一步”只影响对话导航。涉及已激活的学习系统版本时，仍遵循版本化回滚，不直接删除或覆盖历史。

## 8. 显示深度

### 8.1 新手模式（默认）

- 使用短句、通俗选项和单步行动；
- 首屏不显示技术状态；
- 陌生术语首次出现时解释；
- 优先给结果和行动，再按需解释原因。

### 8.2 标准模式

- 显示阶段目标、任务、依据、风险和进度；
- 可以显示能力名称和调整理由；
- 仍隐藏完整结构化状态。

### 8.3 专业模式

- 可显示能力等级、证据引用、版本、闸门和 `engine_result`；
- 不降低证据或安全标准；
- 用户可随时说“简单一点”返回新手模式。

系统可根据用户语言暂时降低术语密度，但不能未经用户同意永久推断其能力或身份。只有用户明确选择时才保存显示深度。

## 9. 异常与边界处理

- 无历史状态：说明尚无可恢复记录，提供“创建新规划”或“指定工作区”两个选择。
- 状态损坏：停止自动写入，展示通俗错误摘要，并遵循现有恢复规则。
- 选择无效：不报技术错误；重述可用选项并允许自然语言回答。
- 请求过于宽泛：显示首页导航，不启动完整 Discovery 问卷。
- 请求明确但证据不足：直接进入对应路径，只问一个最高影响问题。
- 高风险问题：使用 `safety_handoff`，不因界面友好而掩盖风险或越权建议。
- 宿主不支持持久化或资源加载：明确说明当前能力限制，避免声称保存、读取或完成。

## 10. 标识迁移方案

### 10.1 仓库迁移

- 使用 Git 保留历史地把 `learning-architect/` 移动为 `zjskills/`。
- 把 `tests/learning-architect/` 移动为 `tests/zjskills/`。
- 更新脚本说明、测试常量、文档链接、安装命令、平台路径和调用示例。
- 更新 `SKILL.md` 为 `name: zjskills`。
- 更新 `agents/openai.yaml` 默认提示词为 `$zjskills`，显示名保持 `ZJSkills`。
- 更新仓库地址为 `https://github.com/zhijianZJ/ZJSkills.git`。
- 更新 `VERSION` 和中英文版本展示为 `2.0.0`。

历史设计规范和已经完成的实施计划可以保留旧标识作为历史记录，但当前 README、安装文档、运行时、测试命令和贡献指南不得继续把旧标识当作有效调用方式。

### 10.2 用户安装迁移

中英文安装文档提供以下安全流程：

1. 备份旧目录及用户自行修改的内容；
2. 把新版 `zjskills/` 安装到目标平台；
3. 验证 `$zjskills` 或 `/zjskills` 可见并能触发；
4. 验证学习者工作区可继续读取；
5. 确认无误后移走旧 `learning-architect` 安装；
6. 如果验证失败，恢复旧目录并保留错误信息。

不得把新目录直接合并覆盖旧目录，不得同时长期保留两份可发现 Skill。学习者数据结构不因 Skill 目录改名而强制迁移；用户指定的 learner workspace 路径保持不变。

## 11. 文档与版本

需要同步更新：

- `README.md`、`README.en.md`；
- 新手入门、完整使用手册、示例和多平台安装文档的中英文版本；
- `CONTRIBUTING.md`；
- UI 元数据和默认提示词；
- 所有当前验证命令和项目结构表。

README 快速开始应先展示最短调用方式和导航效果，再介绍完整学习系统。使用示例增加模糊请求、数字选择、自然语言选择、“不知道”、继续进度和切换显示深度。

## 12. 测试与验收

### 12.1 命名与迁移测试

- 仓库中存在 `zjskills/SKILL.md`，不存在活动的 `learning-architect/SKILL.md`；
- frontmatter 为 `name: zjskills`；
- `openai.yaml` 使用 `display_name: "ZJSkills"` 和 `$zjskills`；
- 当前公共文档不再给出旧安装路径或旧调用方式；
- 新仓库 URL、2.0.0 版本和跨平台安装路径一致；
- `quick_validate.py zjskills` 通过。

### 12.2 交互行为测试

- 模糊“我想学 AI”显示六项导航，而不是直接生成路线；
- 明确“API 401”直接进入问题卡，不显示首页；
- 回复 `4` 与“我学习卡住了”进入同一路径；
- 回复“不知道”只得到少量选项或极小测试；
- 有有效状态时“继续”从最近动作恢复；
- 无状态时“继续上次”不伪造历史；
- 新手模式不展示 YAML 和内部引擎名；
- 专业模式可以按请求展示结构化状态；
- 一次失败不会自动重建路线；
- 目标变化正确进入 Goal Analysis。

### 12.3 回归验收

- 所有 Schema、模板和有效/无效学习者样例继续通过既有语义校验；
- 中英文相对链接有效；
- 运行时保持无隐藏推广；
- 全部自动化测试通过；
- 新目录安装到本地测试位置后，可通过 `$zjskills` 被发现并完成至少一次模糊入口与一次明确问题的人工验收。

## 13. 发布方式

本次属于不兼容的技术标识迁移，发布为 `2.0.0`。实现提交继续追加到 `codex/learning-architect` 分支和 Draft PR #4；分支名称作为开发历史不要求改名。PR 在自动化测试、安装迁移验证和本地显式调用验证完成后再转为 Ready for Review。
