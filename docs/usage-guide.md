# ZJSkills 完整使用手册

本手册面向希望长期使用 ZJSkills 管理学习与职业成长的人。核心原则是：先定义可验证结果，再设计能力和项目，最后安排内容与时间；学习活动只有在转化为可观察行为和真实证据后，才支持能力判断。

## 完整工作流

阶段按顺序运行。不得因为已经输出了一段文字就越过闸门；确实不适用的阶段必须记录原因、来源、置信度和受影响的下游项。

| 阶段 | 决策目的 | 最小输入 | 主要产物 | 通过条件 | 常用继续提示词 |
|---|---|---|---|---|---|
| 1. Discovery | 建立学习者画像与约束边界 | 目标线索、当前背景、容量、硬约束 | learner profile、SWOT、未知项 | 关键资料有来源和置信度 | “请只问一个最影响路线的问题。” |
| 2. Goal Analysis | 把愿望变成可验证、限时目标 | 目标、期限、成功证据、约束 | target outcome、里程碑、关键结果 | 目标可检查且不与约束静默冲突 | “把我的目标改写为可验证结果。” |
| 3. Gap Analysis | 比较当前证据与目标行为 | 目标能力、当前证据 | 当前/目标等级、优先差距 | 基线有来源，推断与证据分开 | “用现有证据排序最关键差距。” |
| 4. Competency Design | 定义真正需要具备的能力 | 优先差距、岗位/场景参考 | 能力节点、L0-L5 行为、权重、证据要求 | 每个核心节点有目标等级 | “生成可观察的能力树，不要只列知识点。” |
| 5. Curriculum Design | 设计最小且依赖正确的学习图 | 能力差距、先修关系 | learning units、无环依赖图、练习 | 每个单元服务于差距且先修完整 | “按依赖关系设计最小学习图。” |
| 6. Project Design | 用真实项目生产能力证据 | 能力模型、目标场景、约束 | 项目简报、交付物、覆盖矩阵、评分表 | 核心能力均有真实性证据覆盖 | “设计由浅入深的项目和验收标准。” |
| 7. Roadmap | 在现实约束下安排阶段与缓冲 | 项目依赖、容量、预算、期限 | 阶段路线、里程碑、检查点 | 时间、预算和依赖可行 | “给我保守、基准和最低可行路线。” |
| 8. Weekly Planner | 把路线变成一周可兑现承诺 | 当前阶段、可用小时、风险 | 周目标、任务、检索练习、最低交付 | 负载不超容量且能产生证据 | “按本周实际小时生成最低可交付计划。” |
| 9. Assessment | 判断能否独立解决问题 | 交付物、运行结果、评分标准 | 分数、行为观察、证据引用、下一动作 | 关键维度达到证据阈值 | “按证据评分并指出最早因果缺口。” |
| 10. Outcome Preparation | 让目标场景能够评估已有能力 | 已验证证据、目标路线 | 作品集/任务样本/演示/交付材料 | 外部评估者能核验相关证据 | “只用已验证证据准备目标材料。” |
| 11. Continuous Optimization | 根据反馈进行可解释迭代 | 进度、质量、反馈、变化触发 | 诊断、版本变更、预期效果、复盘日 | 变更有原因且受影响产物已复查 | “根据本周数据更新系统并保留历史。” |

多目标竞争同一时间时，必须指定一个主目标；可复用的工作可作为次目标，不兼容目标应延期并设置复查条件。

## 产物与结构化状态

每个阶段返回两层同步结果：

1. 简短自然语言：结论、决定性证据、不确定性和一个下一动作；
2. 统一的 `engine_result`：记录运行、输入、决策、证据、假设、置信度、写入产物、受影响下游、闸门与下一动作。

```yaml
engine_result:
  engine: goal-analysis
  run_id: run-2026-001
  status: needs_input
  summary: 目标方向明确，但成功证据与期限尚未确认
  inputs_used: [learner-statement-001]
  decisions: []
  evidence_refs: []
  assumptions: []
  confidence: low
  artifacts_written: []
  affected_downstream: [gap-analysis]
  gate:
    passed: false
    missing: [deadline, success-evidence]
  next_action: 确认希望在哪个日期用什么产物证明目标已达到
```

`gate.missing` 非空时，`gate.passed` 必须为 `false`。缺少一个会改变决策的答案时使用 `needs_input`；只有存在阻止安全推进的实质障碍才使用 `blocked`。低风险但证据不足时可以创建 `draft`，但必须写明假设和验证动作。

持久化状态以稳定 `id` 和递增 `content_version` 管理。物质性变更产生新版本，旧版本进入 `superseded` 或归档，不直接覆盖历史。继续会话时，从最早未通过闸门恢复，而不是重建已经验证的上游产物。

Skill 不假定一个全局、自动的保存目录。需要跨任务继续时，请在提示词中指定一个你授权的学习者工作区，并要求把入口状态保存为该目录下的 `system-state.yaml`；其他产物按 Schema 约定放在同一工作区或其子目录。若当前工具没有文件写入权限，结果只存在于对话中，`artifacts_written` 必须为空。新任务中应重新提供工作区路径，并要求先读取和验证 `system-state.yaml`。

写文件前应比较已加载版本与磁盘上的活动版本、更新时间或内容哈希。发现外部修改时停止自动覆盖，保留两份状态并请求合并决定。状态损坏时保留损坏副本和恢复轨迹；没有可确定的有效版本时应阻塞，而不是编造状态。

## 证据与能力判断

每条重要信息应同时记录认识论类别和操作来源：

- `fact`：可由稳定外部资料核验的事实；
- `self-report`：学习者自述，例如“会 Python”；
- `evidence`：代码、交付物、测试、评审或观察记录；
- `inference`：由多条信息推导的判断；
- `assumption`：为继续草拟而暂时采用、尚待验证的前提。

来源与置信度必须单独记录。自述可以帮助提出假设，但不能直接证明“能独立交付”。矛盾证据不能静默删除，应降低置信度并安排能够区分结论的最小测评。

以下属于学习活动，不等于能力证据：看完视频、读完书、参加训练、完成打卡、获得证书、感觉理解。能力证据需要观察到目标行为，例如：

- 能否在没有模板修复的情况下独立完成；
- 能否解释关键取舍与限制；
- 能否适应需求变化并修改方案；
- 能否定位和修复故障；
- 能否部署、交付并提供回滚证据；
- 能否向他人讲解或评审。

评估结果应区分 `understanding`、`guided`、`independent` 和 `transfer`，并引用真实证据 ID。项目失败时先定位因果缺口，不应默认增加更多理论内容。

## 版本、复盘与优化

以下事件应触发重新规划：目标变化；容量、预算或设备变化；计划完成率持续偏低；遗忘或迁移能力不足；项目质量或面试表现变化；外部技术、岗位或监管资料过期。

每次变更记录稳定 ID、旧/新版本、触发原因、相关证据、假设、置信度、回滚目标、时间和 `affected_downstream`。变更后重新验证依赖与闸门，并预先定义观察窗口和成功指标，避免因为一次波动频繁改路线。

常用复盘输入包括：计划小时与实际小时、最低交付是否完成、检索练习结果、项目评分、失败类别、外部反馈和约束变化。优化的目标是更高的迁移与交付概率，而不是更高的内容完成率。

## 异常与安全边界

- 可行性冲突：明确指出目标、期限、容量、预算或环境的矛盾，提供取舍或最低可行结果，不承诺外部结果。
- 隐私：只收集影响决策的最少信息；作品、简历、反馈和企业资料应脱敏。不要把密码、密钥、身份证件或未授权数据写入学习状态。
- 高风险内容：求职政策、法律、医疗、财务、生产安全与现实市场判断需要来源日期、限制说明和适当的专业人工复核。
- 外部资料：记录访问日期和适用范围。资料陈旧、相互矛盾或低置信时，返回 `needs_input` 或验证动作。
- 工具与网络：不要声称访问、部署或写入了未实际执行的操作；`artifacts_written` 只能列出真正写入的文件。
- 公共表述：未经证据支持，不把“掌握”“可独立交付”“就业就绪”等结论用于简历或对外材料。
- 外部结果：本 Skill 不能保证录用、晋升、收入、创业、项目验收或其他第三方决定。
